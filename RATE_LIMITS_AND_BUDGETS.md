# Лимиты и бюджеты: как не исчерпать бесплатный tier при 100 пользователях

## Проблема

При 100 пользователях, которые одновременно ведут диалоги с одной моделью (например Groq):

- **Бесплатные лимиты провайдера** (запросов/день, токенов/мин) быстро заканчиваются.
- Один активный пользователь может «съесть» всю квоту, остальные получат ошибки.
- Есть ограничения **на сообщения в минуту, в час** и т.д. у самого провайдера.

Чтобы сервис работал предсказуемо, нужны **ограничения на уровне каждого пользователя** и при желании **общий потолок** на весь прокси.

---

## Механизм в LiteLLM

LiteLLM Proxy умеет выставлять лимиты **на каждый виртуальный ключ** (а у нас один ключ = один пользователь):

| Параметр | Описание |
|----------|----------|
| **rpm_limit** | Запросов в минуту на ключ |
| **tpm_limit** | Токенов в минуту на ключ |
| **max_budget** | Потолок расходов в USD на ключ |
| **budget_duration** | На какой период действует бюджет: `"30d"`, `"1h"`, `"24h"` и т.д. |

При создании ключа через `/key/generate` эти параметры передаются в теле запроса. LiteLLM сохраняет их в БД и при каждом запросе проверяет лимиты. При превышении возвращается ошибка (например 429), а не запрос к провайдеру.

Дополнительно в **config.yaml** можно задать **глобальный бюджет** на весь прокси (`general_settings.max_budget` + `budget_duration`), чтобы суммарный расход всех пользователей не превышал лимит бесплатного tier.

---

## Что реализовано в проекте

### 1. Лимиты при регистрации

При создании ключа для нового пользователя портал передаёт в LiteLLM лимиты из переменных окружения:

- **DEFAULT_MAX_BUDGET** — макс. бюджет в USD на пользователя (например `0.5`).
- **DEFAULT_RPM_LIMIT** — запросов в минуту (например `20`).
- **DEFAULT_TPM_LIMIT** — токенов в минуту (например `10000`).
- **BUDGET_DURATION** — период бюджета (например `30d`).

Если переменные не заданы, лимиты не устанавливаются (ключ без ограничений — удобно для разработки).

### 2. Глобальный потолок (опционально)

В `config.yaml` в `general_settings` можно добавить:

```yaml
general_settings:
  master_key: os.environ/LITELLM_MASTER_KEY
  database_url: os.environ/DATABASE_URL
  # Общий потолок на весь прокси (например $5 на 30 дней)
  # max_budget: 5
  # budget_duration: "30d"
```

Тогда даже если лимиты по ключам не исчерпаны, прокси перестанет пускать запросы после достижения общей суммы.

### 3. Отображение в ЛК

В личном кабинете пользователь видит:

- текущий расход (spend);
- свои лимиты: запросов/мин, токенов/мин, бюджет на период (если они заданы в портале).

Лимиты отдаются с бэкенда из тех же переменных окружения (единый источник правды).

## Рекомендуемые значения для бесплатного tier (Groq и др.)

- **rpm_limit**: 10–30 запросов/мин на пользователя — защита от всплесков и злоупотреблений.
- **tpm_limit**: 5 000–20 000 токенов/мин — зависит от лимитов провайдера (у Groq есть лимиты на RPM).
- **max_budget**: небольшая сумма в USD на пользователя (например 0.5) или 0 при полностью бесплатном tier (тогда ограничиваем только RPM/TPM).
- **budget_duration**: например `30d`, чтобы бюджет считался помесячно.

Имеет смысл свериться с актуальными лимитами провайдера (Groq, OpenAI и т.д.) и выставить **глобальный max_budget** в config.yaml по сумме бесплатного tier.

---

## Ограничения провайдера (RPM/TPM на их стороне)

У провайдера тоже есть лимиты (запросов в минуту, токенов в минуту). Если суммарный трафик всех пользователей превышает их лимит, прокси начнёт получать 429 от API. Варианты:

1. **Задать в LiteLLM лимиты на ключ** (rpm_limit, tpm_limit) так, чтобы `N пользователей × лимит на пользователя` не превышало лимит провайдера.
2. **В config.yaml** для модели задать `tpm`/`rpm` на уровне модели — LiteLLM будет резать трафик к этой модели.
3. По необходимости — очереди, приоритеты, несколько ключей провайдера (advanced).

Итого: механизм — **лимиты на ключ при создании** + опционально **глобальный бюджет** в config; значения подбираются под бесплатный tier и лимиты провайдера.
