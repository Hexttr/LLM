# Архитектура фронтенда и портала для пользователей

## Технологии по образцу jazz2-chi.vercel.app

Сайт размещён на **Vercel** (поддомен `*.vercel.app`). По типичному стеку таких проектов и по названию «chi» (Chakra UI) используется:

| Слой | Технология |
|------|------------|
| **Фреймворк** | Next.js (React) |
| **UI** | Chakra UI («chi» — распространённое сокращение) |
| **Язык** | TypeScript |
| **Хостинг** | Vercel (деплой из Git, serverless) |

Итого для нашего портала: **Next.js + Chakra UI + TypeScript**, деплой на Vercel (или на VPS через Docker/Node).

---

## 1. Список моделей, которые уже доступны у вас

По текущему `config.yaml` и ответу прокси **сейчас доступна одна модель**:

| model_name (для запросов) | Провайдер | Комментарий |
|---------------------------|-----------|-------------|
| **echo** | openai/echo | Тестовая заглушка, возвращает ввод без вызова реального LLM |

Остальные модели в конфиге закомментированы. После добавления в `config.yaml` и перезапуска прокси появятся, например:

- **OpenAI:** `gpt-4o`, `gpt-4o-mini` (нужен `OPENAI_API_KEY` в `.env`)
- **Anthropic:** `claude-3-5-sonnet` (нужен `ANTHROPIC_API_KEY` в `.env`)

Для лендинга список моделей берётся **из `/model/info` через бэкенд**: бэкенд вызывает прокси с мастер-ключом и отдаёт фронту список (model_name и при необходимости описание).

## 3. Блоки функциональности

| Блок | Что делаем | Что откладываем |
|------|------------|------------------|
| **Лендинг** | Секции: о сервисе, список моделей (из API или статичный), тарифы/преимущества, CTA «Начать» / «Регистрация». Стек: Next.js + Chakra UI. | Детальные цены за токен (до появления оплаты). |
| **Регистрация и авторизация** | Регистрация (email + пароль), вход, выход, восстановление пароля (опционально). JWT в httpOnly cookie или заголовке. | OAuth (Google/GitHub) — по желанию позже. |
| **Личный кабинет** | Просмотр/копирование своего API-ключа, отображение баланса и использования (запросы/токены) по данным LiteLLM. | Пополнение и оплата — после переезда на VPS. |

## 4. Принятые решения

| Вопрос | Решение |
|--------|---------|
| **Где хранить пользователей и привязку user → proxy key?** | **БД LiteLLM** — одна Postgres (та же, что у прокси). В ней таблицы LiteLLM (ключи, лимиты) + наши таблицы портала (пользователи: email, хэш пароля, `user_id` для связи с ключом в LiteLLM). |
| **Когда создавать ключ?** | **Сразу при регистрации.** Бэкенд при создании пользователя вызывает `POST /key/generate` с `user_id` и сохраняет выданный ключ (или token_id) у себя / в LiteLLM. |
| **Откуда брать баланс и использование для ЛК?** | **Запросы к API LiteLLM с бэкенда.** Бэкенд по ключу пользователя (или token_id) запрашивает данные у прокси и отдаёт их во фронт. Мастер-ключ только на бэкенде. |
| **Где запускать до переезда на VPS?** | **Всё на одном ПК:** Docker (LiteLLM + Postgres), Next.js (фронт + API routes) локально. См. раздел 6. |
| **Список моделей на лендинге** | **Из `/model/info` через бэкенд.** Бэкенд дергает прокси с мастер-ключом и отдаёт клиенту список моделей (например, только `model_name` и описание). |
| **Тарифы и лимиты** | **Отложить.** Решим после подключения разных LLM. |

## 5. Вопросы, оставшиеся на потом

- **CORS:** при запуске на одном ПК (фронт и бэкенд на localhost) достаточно разрешить запросы с `http://localhost:3000` (или с какого порта крутится Next.js).
- **После переезда на VPS:** фронт и бэкенд можно вынести на VPS или оставить фронт на Vercel; конфиг `NEXT_PUBLIC_API_URL` и `LITELLM_PROXY_URL` вынести в env.

---

## 6. Запуск всего на одном ПК

Пока всё крутится локально на вашем компьютере:

```
Ваш ПК
├── Docker Compose
│   ├── LiteLLM Proxy    → http://localhost:4000
│   └── Postgres         → localhost:5432 (БД общая для прокси и портала)
│
└── Next.js (npm run dev)
    ├── Фронт            → http://localhost:3000 (лендинг, регистрация, ЛК)
    └── API Routes       → http://localhost:3000/api/* (вызовы к LiteLLM с мастер-ключом, работа с пользователями в БД)
```

- **Фронт** обращается только к своему бэкенду: `http://localhost:3000/api/...`.
- **Бэкенд** (Next.js API routes) ходит в LiteLLM по `http://localhost:4000` (или `http://host.docker.internal:4000` из контейнера, если бэкенд позже вынести в Docker) и в Postgres по `localhost:5432`.
- В одной БД Postgres: таблицы LiteLLM + таблицы портала (пользователи, привязка user → key/token_id).

После переезда на VPS те же сервисы поднимаются на сервере, в env задаются реальные URL и секреты.

## 7. Порядок работ

1. **Создать приложение портала:** Next.js + Chakra UI + TypeScript в папке проекта (например, `portal/` или `app/`).
2. **Подключиться к БД LiteLLM:** та же Postgres, добавить миграции для таблиц пользователей (users: id, email, password_hash, litellm_user_id, created_at).
3. **Бэкенд (API routes):** регистрация (создание user в БД + `POST /key/generate` в LiteLLM с этим user_id, сохранение ключа), логин, JWT; эндпоинт списка моделей (`GET /model/info` → прокси); эндпоинт для ЛК (ключ, баланс/использование по ключу из LiteLLM).
4. **Лендинг:** секции + запрос списка моделей с бэкенда.
5. **Личный кабинет:** страница с ключом, балансом и использованием (данные с бэкенда из LiteLLM).
6. **Тарифы и пополнение** — после подключения разных LLM и переезда на VPS.
